API Google Maps Routes API — Geocoding Gap Found

⚠️ Round 2 finding: City-only addresses (e.g. "Chicago, IL") are not being resolved to lat/lng coordinates on the load record. This breaks the broadcast radius check — no offers are sent when pickupLat/pickupLng are null.

  • Load draft geocoding — city name must resolve to lat/lng before submit
  • Broadcast radius check — uses load.pickupLat / load.pickupLng (null = no matches)
  • Tracking display — route overlay on tracking map
What to test / fix

Set GOOGLE_MAPS_API_KEY to an invalid value in EB environment. POST a load draft. Confirm: (1) Returns 200 with null/empty distance, not 500, (2) Load can still be submitted. Also: add a fallback geocoding step in BroadcastService — if load.pickupLat is null, call Maps API to resolve pickupCity + pickupState before running radius filter.

PERF k6 / Artillery — Load & Scale Testing

Section 16 tests require synthetic load generation. These cannot be run from a single curl terminal.

  • Fan-out perf — broadcast to 100+ eligible drivers; measure p95 latency
  • Geohash at scale — geoproximity query as driver count grows to 1,000+
  • DynamoDB burst — on-demand capacity under rapid load-post burst
  • EB autoscaling — verify scale-out triggers before p95 degrades
Recommended tooling

Use k6 with a script that: creates 100 verified driver profiles, posts a load at Chicago coords, submits it, and measures time-to-offer-creation for all 100. Alert if p95 > 2s. Run from AWS EC2 in us-east-1 to avoid client-side network variance.

INFRA DynamoDB Local (Docker) — CI Isolation

The TESTING.md CI gate (Section 17) requires isolated, repeatable test runs. Production DynamoDB cannot be used for CI.

Setup

Run docker run -p 8000:8000 amazon/dynamodb-local in CI. Set DYNAMODB_ENDPOINT=http://localhost:8000 in test env. Run node scripts/createTables.mjs to seed schema. Use Jest + supertest against Express app directly (no HTTP). Authorization matrix + race tests run in <30s.

Real-time driver tracking map on Shipper Load Detail
UI

FEAT
Admin live fleet map
UI
Admin dashboard shows load and driver counts as text. No map showing where all active drivers are. The debug broadcast endpoint already returns lat/lng for every driver — wire it to a Google Maps embed with markers per driver status.
FEAT
Push notification subscription UI for drivers
PUSH
The VAPID endpoint and PushService exist and the backend fan-out sends pushes, but there is no in-app UI prompting the driver to enable browser push notifications. Without a subscription, driver.pushSubscription is null and all pushes silently fail.

In-app notification inbox
UI
Notifications are sent via Resend email and VAPID push but there is no in-app bell/inbox listing past notifications. The header bell icon currently shows a hardcoded orange dot. Wire it to a real notification list (stored in DynamoDB on send).

UI
The new Load History page for driver and OO shows all accepted loads in reverse-accepted order. No date-range filter, no status filter (Booked / In Transit / Delivered), and no pagination. High-volume drivers will see an ever-growing list with no way to narrow it.
FEAT
Carrier analytics dashboard
UI
No revenue or performance metrics for drivers or OOs — total miles driven, earnings by week/month, loads completed, average rate per mile. This data already exists across the loads and offers tables; it just needs a dedicated analytics page.
FEAT
Password change and 2FA in Settings
AUTH

FEAT
POD submission from Load History
UI
Drivers can submit Proof of Delivery via POST /api/driver/loads/:loadId/pod and the POD upload URL endpoint exists, but the Load History page only shows a "View" link to the load detail. The POD upload UI should be accessible directly from the history list for loads in BOOKED / IN_TRANSIT status.

P2
Stale push subscription pruning
PUSH
If a driver's browser push subscription expires or the device changes, PushService.send() will receive a 410 Gone response from the push server. This is not currently handled — it logs an error and continues. Expired subs should be deleted from LoadLead_PushSubscriptions on 410/404 response.
P2
File upload content-type allowlist (POD photos)
SEC
The presigned S3 URL for POD photo upload specifies ContentType: fileType from the request body — but fileType is not validated. A caller could request a presigned URL for application/x-executable and upload a binary file. Restrict fileType to ['image/jpeg','image/png','image/webp'] in the route handler.
P2
Token expiry mid-session → auto-redirect to login
AUTH
When the 7-day JWT cookie expires mid-session, the next API call returns 401. The frontend currently surfaces this as a toast error ("GET /driver/loadboard failed (401)") rather than redirecting to /login. Add a 401 interceptor in api.ts that calls logout() and navigates to /login.
